# Инструкция для агента: внедрение режима «супер-качество» (QUALITY_MODE)

Документ — пошаговое ТЗ для реализации в `scripts/agents/pipeline.py` (и связанных файлах
`registry.py` / конфиг моделей). Цель: добавить опциональный премиум-режим генерации статей
БЕЗ поломки текущего экономного режима. Каждый шаг автономен и имеет критерии приёмки.

ВАЖНЫЕ ОГРАНИЧЕНИЯ (соблюдать строго):
- Это юридические статьи. Лаконичность и плотность — это ПЛЮС. НЕ смягчать и НЕ менять
  `density_config` / density-пресеты. Никакой «воды», примеров ради объёма, лирики.
- Все новые тяжёлые шаги включаются ТОЛЬКО под флагом QUALITY_MODE. По умолчанию (флаг выключен)
  поведение пайплайна не меняется.
- Любой цикл правок ДОЛЖЕН иметь жёсткий потолок итераций и стратегию «accept-best», чтобы
  система не ходила по кругу.
- Правки текста — ТОЛЬКО посекционные (точечные), переписывание всей статьи целиком запрещено,
  кроме явного аварийного фолбэка.

---

## ШАГ 0. Флаг режима QUALITY_MODE

Цель: единая точка включения премиум-логики.

Действия:
1. Ввести чтение флага в начале `run()` пайплайна:
   - `quality_mode = os.getenv("QUALITY_MODE", "false").lower() in ("1","true","yes","on")`
   - либо параметр функции `run(..., quality_mode: bool = False)`, а env — как дефолт.
2. Сохранить в state: добавить поле `quality_mode: bool = False` в датакласс `PipelineState`,
   присвоить при инициализации state.
3. В лог при старте: `logger.info(f"Режим качества: {'ВКЛ' if quality_mode else 'выкл'}")`.

Критерий приёмки: при `QUALITY_MODE=false` ни один новый шаг не выполняется; диффы поведения = 0.

---

## ШАГ 1. Реестр моделей: добавить внешнего ревизора (Gemini / Claude через kie.ai)

Цель: дать возможность судить текст ДРУГОЙ моделью, не той, что писала.

Действия:
1. В конфиг моделей (`MODELS` dict / `registry.py`) добавить запись, например:
   - `"reviewer_external": { provider: "kie", model: "gemini-3-pro" (или "claude-..."), ... }`
2. Реализовать/расширить клиент для kie.ai (если ещё нет): функция, принимающая
   `(system_prompt, user_msg)` и возвращающая текст/JSON. Ключ API — из окружения
   (`KIE_API_KEY`), НЕ хардкодить.
3. Зарегистрировать агента-ревизора `external_reviewer` с этой моделью и системным промптом
   (см. ШАГ 4). Агент работает в JSON-режиме (`parse_json=True`).
4. Учитывать токены этого агента в общем счётчике `tokens.by_agent` (как у остальных).

Критерий приёмки: вызов `_call_agent("external_reviewer", msg, parse_json=True)` возвращает
валидный JSON; токены попадают в отчёт; при отсутствии `KIE_API_KEY` режим деградирует
безопасно (логируем предупреждение и пропускаем внешний ревизор, не падаем).

## ШАГ 2. Критика плана статьи ДО написания (Engineer → Critic → Engineer)

Цель: чинить структуру дёшево, до генерации текста. Самый высокий ROI по качеству.

Действия (выполнять только при `quality_mode`):
1. После `_step_engineer(state)` (построение Blueprint) добавить шаг
   `_step_plan_critique(state)`:
   - Вход: `state.blueprint` (JSON структуры), тема, целевой объём, ключи.
   - Вызов критика (можно `external_reviewer` или отдельный промпт на основной модели) с
     требованием вернуть JSON:
     ```json
     {
       "ok": false,
       "issues": [
         {"section": "H2 заголовок или индекс", "problem": "...", "fix": "..."}
       ],
       "missing_topics": ["..."],
       "redundant_sections": ["..."]
     }
     ```
   - Если `ok=false` — один раз пересобрать Blueprint Engineer'ом с учётом замечаний
     (передать `issues` в `retry_msg`). Максимум 1 повтор (без цикла).
2. Логировать, что план был улучшен; сохранить финальный Blueprint в state.

Критерий приёмки: при кривом плане Blueprint меняется один раз; H2-разделы покрывают
`missing_topics`; дубли убраны. Потолок: 1 итерация критики.

## ШАГ 3. Best-of-N черновиков + выбор лучшего (замена тяжёлого revise-loop)

Цель: вместо «черновик + N полных переписей по кругу» — сгенерировать несколько независимых
черновиков и ВЫБРАТЬ лучший. Часто это дешевле и качественнее.

Действия (только при `quality_mode`):
1. Параметр `N = int(os.getenv("QUALITY_DRAFTS", "2"))` (по умолчанию 2, допустимо 3).
2. В `_step_heart` (или новой обёртке `_step_heart_best_of`) сгенерировать N независимых
   черновиков:
   - Каждый — отдельный вызов существующего пути генерации (`_heart_single` / `_heart_sectional`).
   - Черновики независимы (без общего накопленного фидбека).
   - Желательно параллельно (ThreadPool), если клиент потокобезопасен; иначе последовательно.
3. Оценить каждый черновик ОДНИМ прогоном Sheriff (и/или external_reviewer): получить
   `turing_score` + замечания.
4. Выбрать черновик с лучшим баллом → это `state.draft`.
   - (Опционально, фаза 2) Гибрид: собрать статью из лучших H2-секций разных черновиков по
     посекционным оценкам. Делать ТОЛЬКО если секции совместимы по стилю; иначе — просто лучший.
5. ВАЖНО: в quality_mode best-of-N ЗАМЕНЯЕТ старый revise-loop `for i in range(2)` —
   не запускать оба, иначе двойной расход.

Критерий приёмки: при N=2 генерируется ровно 2 черновика, выбирается один с макс. баллом;
старый revise-loop в quality_mode не выполняется.

## ШАГ 4. Финальный ревизор другой моделью + посекционный план правок

Цель: «второе мнение» от Gemini/Claude, которое НЕ переписывает текст, а выдаёт точечные
замечания по секциям.

Действия (только при `quality_mode`, после выбора лучшего черновика):
1. Вызвать `external_reviewer` с системным промптом «ты строгий редактор юридических статей,
   ценишь точность, лаконичность и фактическую корректность; стиль — деловой, без воды».
2. Передать: тему, целевой объём, и текст статьи, РАЗБИТЫЙ на секции (используй существующий
   `_split_markdown_sections`), с индексами секций.
3. Требовать строго JSON:
   ```json
   {
     "approved": false,
     "score": 0,
     "edits": [
       {"section_index": 2, "instruction": "что именно исправить в этой секции"}
     ]
   }
   ```
   Правило: `instruction` — конкретное и локальное (внутри одной секции). Запрещено просить
   «переписать всю статью».
4. Если `approved=true` или `score >= ПОРОГ` (например 85) — выходим, правок нет.

Критерий приёмки: ревизор возвращает список `edits` с валидными `section_index`; при отсутствии
ключа kie.ai шаг безопасно пропускается.

## ШАГ 5. Исполнитель правок: ТОЛЬКО точечно + предохранители от зацикливания

Цель: применять замечания (от Sheriff/Mirror/external_reviewer) к КОНКРЕТНЫМ секциям, не трогая
остальные, и гарантированно не ходить по кругу.

База: у вас уже есть `_heart_patch`, `_build_edit_plan`, `_rewrite_section`, `_section_ok`,
`_split_markdown_sections`, `_reassemble_sections`. Их и расширяем.

Действия:
1. Единый цикл доводки в quality_mode:
   ```
   approved_sections = set()      # секции, которые УЖЕ одобрены — не трогаем
   best_draft, best_score = state.draft, -1
   for it in range(MAX_QUALITY_ITERS):     # MAX_QUALITY_ITERS = 3 (жёсткий потолок)
       review = collect_review(state)      # Sheriff + (если low) Mirror + external_reviewer
       score = review.score
       if score > best_score:              # accept-best: запоминаем лучший вариант
           best_score, best_draft = score, state.draft
       if review.approved or score >= THRESHOLD:
           break
       edits = [e for e in review.edits if e.section_index not in approved_sections]
       if not edits:
           break
       for e in edits:
           rewrite ONLY section e.section_index via _rewrite_section(...)
           if _section_ok(...):
               mark section ok; if section passed review -> approved_sections.add(idx)
       # защита структуры: если H2 пропали или объём упал <60% — откат секции к оригиналу
   state.draft = best_draft if not approved else state.draft   # accept-best на выходе
   ```
2. Три обязательных предохранителя:
   - ЖЁСТКИЙ ПОТОЛОК: `MAX_QUALITY_ITERS = int(os.getenv("QUALITY_MAX_ITERS","3"))`.
   - ПАМЯТЬ ПО СЕКЦИЯМ: одобренную секцию в следующих итерациях НЕ перепроверять и НЕ переписывать
     (иначе «починили A — сломали B — чиним B — сломали A»).
   - ACCEPT-BEST: если за потолок итераций не достигли THRESHOLD — берём вариант с лучшим баллом,
     а не крутим бесконечно и не отдаём заведомо худший последний.
3. Запрет полной переписи: переписывание всей статьи (`_step_combined_revision`) допускается ТОЛЬКО
   как аварийный фолбэк, когда помечено >50% секций — и логируется как WARNING.

Критерий приёмки: на «упрямой» статье цикл гарантированно завершается за ≤ MAX_QUALITY_ITERS;
итоговый текст — не хуже лучшего из встреченных (accept-best); одобренные секции не дёргаются.

## ШАГ 6. Посекционный QA для маленьких статей (в связке с посекционной правкой)

Цель: распространить посекционный контроль (сейчас он только для больших) на маленькие статьи,
но строго вместе с посекционной правкой из ШАГА 5 — чтобы брак одной секции не тянул переписывание
всей статьи.

Действия (только при `quality_mode`):
1. Снять/понизить порог, при котором включается посекционная обработка (сейчас в `_step_heart`
   секционный путь идёт при `heart_target > 20000`). В quality_mode разрешить секционный QA и для
   маленьких статей — НО правки применять только к бракованным секциям (ШАГ 5).
2. Убедиться, что `_section_ok` адекватно работает на коротких секциях (не заваливает нормальные).

Критерий приёмки: маленькая статья с одной слабой секцией чинится точечно; остальные секции
байт-в-байт сохранены.

---

## ШАГ 7. Роль «редактор-стилист» — опционально (если захотите)

Контекст: сейчас Sheriff и Mirror — это КРИТИКИ-судьи (возвращают вердикт/score, текст не правят).
Отдельной роли «редактор, который сам улучшает текст» нет.

Действия (опционально, только quality_mode):
1. Добавить агента `editor`, который НЕ судит, а делает финальную посекционную шлифовку
   (ритм предложений, переходы, убрать канцелярит/повторы) БЕЗ изменения смысла и фактуры и БЕЗ
   раздувания объёма (юридический стиль — лаконичность сохранять).
2. Применять посекционно (ШАГ 5), один проход, после прохождения ревизоров.

Критерий приёмки: объём не вырос значимо; H2/факты/ссылки на месте; стиль ровнее.

## Переменные окружения (сводно)

- `QUALITY_MODE` (true/false) — включает премиум-режим.
- `KIE_API_KEY` — ключ kie.ai для внешнего ревизора.
- `QUALITY_DRAFTS` (по умолч. 2) — число черновиков для best-of-N.
- `QUALITY_MAX_ITERS` (по умолч. 3) — жёсткий потолок итераций доводки.
- (порог одобрения THRESHOLD — константа в коде, напр. 85.)

## Порядок шагов в quality_mode (итоговый конвейер)

Brain → Fact-Finder → Freshness → Scout → Engineer → **[ШАГ 2: критика плана]** →
**[ШАГ 3: best-of-N черновиков + выбор]** → **[ШАГ 4: внешний ревизор → план правок]** →
**[ШАГ 5/6: посекционная доводка с предохранителями]** → **[ШАГ 7: редактор (опц.)]** →
Booster → smart_hard_cut → Artist.
